# Fork Operations

Notes for maintaining a fork of litter that builds and ships to TestFlight
independently of upstream. Covers the non-obvious traps hit while setting
up the `cad0p/litter` fork; most of this is not documented elsewhere.

## Bundle ID rebrand

Upstream uses `com.sigkitten.litter`. Forks **must** rebrand — App Store
Connect bundle IDs are globally unique across all developer accounts, so
you cannot register `com.sigkitten.litter` under your own account.

Use the included tool to rebrand across iOS + Android in one pass:

```bash
./tools/scripts/switch-app-identity.sh --to your-identifier --identifier <name>
```

This rewrites `com.sigkitten.litter` → `com.<name>.litter` in all tracked
files **except** `apps/ios/Litter.xcodeproj/` (regenerated by `make xcgen`)
and the script itself. If you need a non-`com.` prefix (e.g. `it.pcad.litter`),
run the equivalent `perl -0pi -e 's/com\.sigkitten\.litter/it.pcad.litter/g'`
over `git grep -zl` results, skipping those same two paths.

After rebranding, also update the hardcoded `APP_BUNDLE_ID` /
`LIVE_ACTIVITY_BUNDLE_ID` in `.github/workflows/ios-testflight.yml` (the
script defaults read from env, but the workflow sets them explicitly).

## App Store Connect CLI (`asc`)

Install: `curl -fsSL https://asccli.sh/install | bash` (Go binary, runs on
Linux). Authenticate with an App Store Connect API key (App Manager role):

```bash
asc auth login --bypass-keychain --name devbox \
  --key-id <KEY_ID> --issuer-id <ISSUER_ID> \
  --private-key /path/to/AuthKey.p8 --network
```

`asc` can create **almost everything** a fork needs via the public API:
- App IDs: `asc bundle-ids create --identifier <bundle> --name <name> --platform IOS`
- Capabilities: `asc bundle-ids capabilities add --bundle <id> --capability APP_GROUPS`
- Certificates: `asc certificates csr generate` then `asc certificates create --certificate-type IOS_DISTRIBUTION --csr ./cert.csr`
- Profiles: `asc profiles create --name <name> --profile-type IOS_APP_STORE --bundle <id> --certificate <cert_id>`
- Profile download: `asc profiles download --id <id> --output ./profile.mobileprovision`

### The App Group gap (important)

The public App Store Connect API **cannot assign an app group container to
a bundle ID** — it can only enable the `APP_GROUPS` *capability*. Profiles
generated via `asc profiles create` come out with an **empty**
`com.apple.security.application-groups` array even if you linked the group
in the Developer Portal web UI. The archive then fails with:

```
Provisioning profile "..." doesn't support the group.<x> App Group.
```

**Fix**: use **fastlane `sigh`** to download profiles — it reads the
portal's true state (including app group linkage) and produces correct
profiles. fastlane uses the same API key:

```bash
gem install fastlane --no-document
# api_key.json must embed the .p8 contents (not a path):
python3 -c "
import json
key = open('/path/to/AuthKey.p8').read()
json.dump({'key_id':'<KEY_ID>','issuer_id':'<ISSUER_ID>','key':key,'in_house':False},
          open('api_key.json','w'))
"
fastlane run sigh api_key_path:"api_key.json" \
  app_identifier:"<bundle>" readonly:true skip_install:true \
  output_path:"." filename:"<name>.mobileprovision"
```

Verify the profile includes the app group before uploading as a secret:
```bash
openssl smime -inform DER -verify -in <name>.mobileprovision -noverify 2>/dev/null \
  | grep -A2 'application-groups'
# should show: <string>group.<x></string>  (not an empty <array></array>)
```

### Capabilities not in the public API

Some entitlements require special Apple approval and are **not** in the
public API capability list (so `asc bundle-ids capabilities add` rejects
them). Known: `com.apple.developer.carplay-voice-based-conversation`.

If your fork doesn't have that approval, remove the entitlement from
**both** `apps/ios/Sources/Litter/Litter.entitlements` **and**
`apps/ios/project.yml` (the `properties:` block under the entitlements
path). `make xcgen` regenerates the `.entitlements` file from
`project.yml`, so editing only the `.entitlements` file is undone on the
next project regen — this is an easy trap.

## Distribution certificate .p12 format

macOS `security import` rejects PKCS12 files produced by modern openssl 3.x
with `SecKeychainItemImport: MAC verification failed during PKCS12 import
(wrong password?)`. The password is correct — the issue is the MAC
algorithm (SHA-256 / AES-256-CBC) that macOS `security` doesn't understand.

Regenerate the .p12 with the **legacy** flag so macOS accepts it:

```bash
openssl pkcs12 -export -legacy \
  -inkey dist.key -in dist.cer -out dist.p12 \
  -passout "pass:$PASSWORD" -name "Litter Distribution"
```

## TestFlight build workflow

### sccache on forks without Cloudflare R2

The upstream workflow assumes `SCCACHE_R2_*` secrets exist. On forks
without R2, `build-rust.sh` sources `load-sccache-aws-creds.sh` which sets
`AWS_PROFILE=cloudflare-r2-sccache` and a hardcoded `SCCACHE_ENDPOINT`.
The `aws-sdk` crate (transitive dep via codex) then sees `AWS_PROFILE`,
finds no local creds, and falls back to the **EC2 instance metadata
endpoint** at `169.254.169.254` — which fails on macOS CI runners with
`No route to host`, failing the entire Rust cross-compile.

`build-rust.sh` now guards the sourcing with `SKIP_SCCACHE_AWS_CREDS=1`,
and `ios-testflight.yml` sets that + `CARGO_INCREMENTAL=1` in the no-R2
branch (the latter disables `build-rust.sh`'s own `RUSTC_WRAPPER=sccache`
auto-enable guard). Local disk sccache only helps within a single job run
(reruns), not across runs, so skipping it entirely without R2 is the right
call.

### Rust dependency cache

`Swatinem/rust-cache@v2` is wired into the workflow with
`workspaces: "shared/rust-bridge -> target"` and `save-if: ${{ github.ref
== 'main' }}`. The cache is ~819 MB (measured from the prep artifact),
well under GitHub's 10 GB limit. First run is slow (~50 min Rust
cross-compile for `aarch64-apple-ios`); subsequent runs with a warm cache
are faster for dependency crates but the final link + the
`codex-mobile-client` crate itself still compile, so expect ~30-40 min
even with cache.

### Standalone upload workflow

`.github/workflows/ios-testflight-upload.yml` is a standalone
`workflow_dispatch` workflow that downloads the `ios-release-prep`
artifact from a prior successful `iOS TestFlight Release` run (via the
`prep_run_id` input, or auto-finds the latest successful run) and runs
only the Xcode archive + TestFlight upload. Use this to retry the upload
step (~10-15 min) without rebuilding Rust (~50 min) when the upload fails
for transient reasons (network blips, missing beta metadata, etc.):

```bash
gh workflow run ios-testflight-upload.yml --ref main -R <owner>/litter \
  -f prep_run_id=<prior-successful-run-id> \
  -f beta_group_names="Internal Testers"
```

### Internal vs external TestFlight groups

The upload step submits for **Beta App Review** when an external group is
in `BETA_GROUP_NAMES`. External review requires `betaAppLocalizations`
and `betaAppReviewDetails` to exist on the app record. For quick
iteration, use internal-only:

```bash
-f beta_group_names="Internal Testers"
```

Internal testing needs no review and installs immediately. To set the
required beta metadata for external testing (via `asc`):

```bash
asc testflight app-localizations create --app <APP_ID> \
  --locale en-US --description "<description>"
asc testflight review edit --id <APP_ID> \
  --contact-email <email> --contact-first-name <first> \
  --contact-last-name <last> --contact-phone "+CC.NNNNNNNNNN"
```

The phone format must be `+CC.NNNNNNNNNN` (dot separator, E.164-ish) —
other formats are rejected as invalid.

## Required secrets (11)

Set these in the fork's repo secrets (Settings → Secrets → Actions):

| Secret | Source |
|---|---|
| `ASC_KEY_ID` | App Store Connect API Key ID |
| `ASC_ISSUER_ID` | Issuer UUID (ASC Keys page) |
| `ASC_PRIVATE_KEY_P8_B64` | `base64 -w0 AuthKey_XXXX.p8` |
| `IOS_TEAM_ID` | 10-char Apple Developer Team ID |
| `IOS_APP_STORE_APP_ID` | numeric ASC app ID (from app URL) |
| `IOS_DIST_CERT_P12_B64` | `base64 -w0 dist.p12` (legacy format!) |
| `IOS_DIST_CERT_PASSWORD` | .p12 export password |
| `IOS_APP_STORE_PROFILE_B64` | App profile, `base64 -w0` (via fastlane sigh) |
| `IOS_LIVE_ACTIVITY_APP_STORE_PROFILE_B64` | Live Activity profile |
| `IOS_WATCH_APP_STORE_PROFILE_B64` | Watch app profile |
| `IOS_WATCH_COMP_APP_STORE_PROFILE_B64` | Watch complications profile |

Optional (speed up Rust builds via remote cache): `SCCACHE_R2_ENDPOINT`,
`SCCACHE_R2_ACCESS_KEY_ID`, `SCCACHE_R2_SECRET_ACCESS_KEY` (Cloudflare R2).

## Commit signing

Commits are SSH-signed. The signing key must be registered on GitHub as a
**Signing Key** (not Authentication). `gh ssh-key add --type signing` needs
the `admin:ssh_signing_key` scope — run `gh auth refresh -h github.com -s
admin:ssh_signing_key` first. Local verification needs an allowed-signers
file:

```bash
echo "<email> $(cat ~/.ssh/signing_key.pub)" > ~/.config/git/allowed_signers
git config --global gpg.ssh.allowedSignersFile ~/.config/git/allowed_signers
```
